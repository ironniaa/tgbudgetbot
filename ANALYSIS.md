# Аналитика проекта TGBudgetBot

Дата анализа: 2026-07-03

Документ описывает текущее состояние функциональности бота, найденные проблемы и список работ, необходимых для того, чтобы начать пользоваться ботом на повседневной основе («релиз» для внутреннего использования двумя людьми).

---

## 1. Что это за бот

Telegram-бот для учёта личных и семейных расходов на двух человек (whitelist из `FIRST_ID`/`SECOND_ID`). Данные пишутся в SQLite и синхронно зеркалируются в Google Sheets. Два способа ввода траты:

1. **Guided flow (FSM)**: `/start` → выбор режима «Личное»/«Общее» → категория (inline-кнопки) → сумма → комментарий → сохранение.
2. **Quick input**: одна строка вида `<алиас> <сумма> [комментарий]`, категория определяется по словарю алиасов.

Дополнительные команды: `/reset` (сброс состояния), `/history` (последние 10 операций), `/undo` (удалить последнюю свою запись).

## 2. Архитектура и стек

- Python 3.10, `python-telegram-bot` (polling, без webhook), SQLAlchemy 2.0, SQLite (`database.db`), Alembic для миграций, `gspread` + `google-auth` для Google Sheets.
- Хранение: единственная таблица `expenses` (`id, created_at, timezone, creator, owner, category, amount, comment, synced`). Нет таблиц пользователей/категорий — всё денормализовано строками.
- Access control: белый список из ровно двух Telegram ID в `bot/config.py`, проверяется через `check_access()` в начале хендлера.
- Синхронизация с Google Sheets происходит синхронно и eager — сразу после каждой записи расхода вызывается `sync_unsynced_expenses()`, которая пушит все ещё не синхронизированные строки.
- Миграции Alembic (2 штуки) применяются автоматически при каждом старте бота (`init_db()` вызывает `alembic upgrade head`).

Это уже рабочий MVP: базовый цикл «записал трату → трата попала в Sheets» функционирует. Но по инфраструктурной зрелости (логи, ошибки, тесты, деплой) проект на уровне прототипа.

## 3. Сильные стороны текущей реализации

- Базовый happy path (запись расхода в обоих режимах ввода) работает и покрывает обе целевые механики: быстрый ввод и guided flow.
- Alembic-миграции — наиболее зрелая часть проекта: две миграции, идемпотентные, с защитой от повторного применения, автоприменяются при старте.
- Access control реализован и применяется в большинстве хендлеров.
- Простая, понятная модель данных без избыточной сложности.

## 4. Найденные проблемы

### 4.1 Критично — блокирует повседневное использование

- **Секреты лежат в открытом виде на диске**: реальный `BOT_TOKEN`, `FIRST_ID`/`SECOND_ID`, а также `credentials/google.json` с приватным ключом service account — всё это в рабочей директории проекта (не в git, но без какой-либо защиты/шаблона). Нужно завести `.env.example` без реальных значений и убедиться, что процесс ротации токена/ключа при компрометации понятен.
- **`requirements.txt` практически бесполезен** — содержит только `alembic`, хотя реально используются `python-telegram-bot`, `python-dotenv`, `SQLAlchemy`, `gspread`, `google-auth`. Окружение невозможно воспроизвести командой `pip install -r requirements.txt`. Это прямой риск при переустановке/переносе на новый сервер.
- **Нет процесс-менеджера/автозапуска**: бот запускается вручную (`python main.py`) через polling. Нет systemd unit / docker-compose / любого supervisor. При падении процесса, перезагрузке сервера или обрыве сети бот просто не работает — для «использования на ежедневной основе» это ключевой пробел.
- **Синхронизация с Google Sheets без обработки ошибок**: если Google API временно недоступен (сеть, rate limit, невалидный токен), исключение всплывает необработанным прямо в хендлере. Расход уже сохранён в SQLite (commit случился раньше), но пользователь получит ошибку/отсутствие ответа и решит, что трата не сохранилась — либо повторно её введёт (дублирование). Нужен try/except вокруг `sync_unsynced_expenses()` с понятным сообщением пользователю независимо от результата синхронизации.
- **Нет глобального error handler** (`app.add_error_handler`) — любое необработанное исключение в хендлере просто «роняет» обработку конкретного обновления без ответа пользователю и без структурированного лога.
- **Нет логирования** (единственный `print("Bot started")`). При сбое в проде невозможно понять, что произошло — ни ошибок доступа, ни ошибок синхронизации, ни действий пользователей не фиксируется.

### 4.2 Важно — стоит закрыть до релиза, но не блокирует физическую работу бота

- **Несогласованный access control**: хендлеры `/reset` и обработчик inline-кнопок категорий (`callbacks.py`) не вызывают `check_access()`, в отличие от всех остальных. Нужно унифицировать.
- **Bare `except:` в `text.py`** в двух местах, один из них (Quick input) молча проглатывает нераспознанный ввод без ответа пользователю — с точки зрения UX это выглядит как «бот не отвечает», непонятно что не так.
- **Несогласованная таксономия категорий**: три разных источника категорий не совпадают — Quick input содержит «уход», guided flow (inline-кнопки) содержит «здоровье» вместо «уход», а неиспользуемый (мёртвый) reply-keyboard код не содержит ни то ни другое. Это создаёт разные наборы возможных значений `category` в БД в зависимости от способа ввода и усложнит будущую агрегацию расходов по категориям.
- **`/history` и Google Sheets не разделяют данные по пользователю/режиму**: `/history` показывает последние 10 операций вообще без фильтра (ни по owner, ни по creator, ни по периоду), оба пользователя видят весь поток, включая «Личное» другого человека.
- **Нет функции сумм/статистики** — вообще нет команды показать «сколько потрачено за месяц/неделю/по категориям». Всё, что есть — сырой список последних 10 операций. Для регулярного использования семейного бюджета обычно нужен минимум «сколько потратили в этом месяце» и «разбивка по категориям».
- Db-сессии закрываются без `try/finally` — при исключении между открытием и `db.close()` соединение может не закрыться (утечка, некритично при малой нагрузке, но плохая практика).
- ID Google Sheets-таблицы захардкожен в коде, а не в `.env` — смена таблицы требует правки кода и деплоя.

### 4.3 Второстепенно — можно отложить

- Нет тестов вообще (ни unit, ни integration) — рискованно для денежных данных, но при штате «два пользователя, ручное использование» не блокирует релиз.
- Мёртвый код: `categories_keyboard()` в `bot/utils/keyboards.py` определена и импортируется, но никогда не вызывается — стоит удалить.
- README и CLAUDE.md частично устарели (упоминают несуществующий `bot/handlers/category.py`, список «планируемых функций» включает уже реализованные `/history`/`/undo`/inline-кнопки).
- `alembic.ini` хранит `sqlalchemy.url` отдельно от `bot/database/database.py` — два места с одинаковым значением, при изменении пути к БД легко забыть обновить одно из них.
- Нет CI (тесты/линт при пуше) — нормально для проекта такого масштаба, но полезно завести хотя бы простой lint-check.

## 5. Что нужно сделать, чтобы отправить бота в повседневное использование

Минимальный набор, без которого использовать бот на ежедневной основе рискованно или неудобно:

1. **Восстановить `requirements.txt`** (`pip freeze` из текущего `.venv`, вычистив служебные пакеты) — чтобы окружение можно было переставить.
2. **Добавить обработку ошибок синхронизации с Google Sheets** — try/except вокруг вызова `sync_unsynced_expenses()`, ответ пользователю в духе «трата сохранена, синхронизация с таблицей отложена» вместо необработанного исключения.
3. **Добавить `app.add_error_handler()`** в `main.py` — единая точка перехвата, логирование traceback, нейтральный ответ пользователю («что-то пошло не так, попробуйте позже»).
4. **Настроить базовое логирование** (модуль `logging`, вывод в файл или stdout с ротацией) — минимум: старт бота, ошибки, факты успешной/неуспешной синхронизации.
5. **Обеспечить автозапуск/устойчивость процесса** — systemd unit (или аналог: pm2, screen+cron watchdog, Docker + restart policy) на сервере, где бот будет жить постоянно.
6. **Унифицировать access control** — добавить `check_access()` в `/reset` и в обработчик callback-кнопок.
7. **Создать `.env.example`** с плейсхолдерами (без реальных значений) и убедиться, что реальные `.env`/`credentials/google.json` не теряются при переносе (например, задокументировать процесс их создания заново на новом сервере, или подготовить безопасный процесс передачи).
8. **Свести таксономию категорий к одному источнику правды** — унифицировать «здоровье»/«уход» между Quick input и guided flow, убрать мёртвый reply-keyboard код.

Желательно, но можно сделать после первого релиза (быстрые доработки для удобства ежедневного использования):

9. Добавить простую команду статистики (например `/stats` — сумма за текущий месяц, разбивка по категориям, отдельно «Личное» каждого и «Общее»).
10. Отфильтровать `/history` по режиму/пользователю или добавить параметры (`/history личное`, `/history общее`).
11. Заменить bare `except:` на конкретные исключения; для Quick input добавить ответ пользователю при нераспознанном вводе вместо молчания.
12. Обновить README/CLAUDE.md, убрать упоминания несуществующих файлов и уже реализованных «планируемых» фич.

Не обязательно для повседневного использования двумя людьми (можно отложить надолго): тесты, CI/CD, Docker, поддержка более 2 пользователей, OCR чеков, AI-парсинг, дашборды — это уже развитие продукта, а не условие для начала использования.
